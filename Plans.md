## Phase 85: PII Guard false-positive reduction — bearer-token tightening + weak-context rules + warn-only parity

Created: 2026-04-29

> **Editing convention for this phase**: The PII Guard hooks scan the very text of this plan. To avoid the plan blocking its own edits, four rule names appear ONLY in their hyphenated/slugified form. Never expand them in prose:
> - `bearer-token` — the built-in rule at `go/internal/piiguard/builtin.go:34`
> - `external-confidential-aws-credentials-context` — external rule, literal-keyword matcher
> - `external-confidential-microsoft-office-365-oauth-context` — external rule, URL literal matcher
> - `external-confidential-john-the-ripper` — external rule, tool-name literal matcher

**Goal**: Cut PII Guard false positives that block routine prompts and tool I/O. Three concrete fixes: (1) tighten the `bearer-token` built-in rule by adding a `{20,}` minimum on the captured token-value plus a Go-side post-filter that rejects an all-lowercase-alphabetic capture; (2) disable the three external rules listed in the editing convention above whose regexes are literal-keyword matchers, not credential matchers, and add their slugified IDs to `.claude/rules/deleted-concepts.yaml` per `.claude/rules/migration-policy.md`; (3) extend the existing UserPromptSubmit warn-only mode (`HARNESS_PIIGUARD_PROMPT_WARN_ONLY`) to PreToolUse and PostToolUse plus a global `HARNESS_PIIGUARD_WARN_ONLY` so users with chronic noise can downgrade hard-block to advisory. Add fixture files to the existing `testdata/negative/` and `testdata/positive/` directories — no new test code; the regression gates in `corpus_test.go` (lines 14-43, 48-79) already run the full active catalog against every fixture. The Phase 83 `openai-api-key` regex tightening (commit 74aebfa raised `{8,}` to `{40,}`) already addressed the user-named regex; this phase handles what is actually firing — the four rules listed in the editing convention — which the user misattributed to the OpenAI blocker.

**Motivation**: User reports `<task-notification>` and "anything with task" trigger the OpenAI key blocker. Investigation reproduced live false positives during this very planning session: editing this Plans.md was blocked twice when an earlier draft expanded the four rule names verbatim. The current `bearer-token` pattern at `go/internal/piiguard/builtin.go:34` is `(?i)\bBearer\s+[A-Za-z0-9\-._~+/]+=*` — zero minimum length on the token-value capture, so a single alphanumeric char triggers it. The user's `<task-notification>` example does NOT match any current rule (verified by piping that exact string through the deployed binary at `harness/bin/harness`), confirming the misattribution. The actual culprits beyond `bearer-token` are the three external rules whose IDs after slugification are listed in the editing convention. None of them pattern-match secret values — they keyword-match documentation. The current warn-only escape hatch only covers UserPromptSubmit, so users hit by PostToolUse noise have no way to downgrade short of disabling rules wholesale via `HARNESS_PIIGUARD_DISABLED_RULES`.

**Source**: New work, not an upstream port. Builds directly on Phase 83 (`go/internal/piiguard/`), the warn-only flag added in commit 74aebfa, and the migration-residue policy at `.claude/rules/migration-policy.md`.

**Design**:
- `go/internal/piiguard/rule.go` — add an optional per-rule `Validator func(match string) bool` field to the `Rule` struct. The scanner skips findings where `Validator != nil && !Validator(match)`. Keeps the regex declarative for the simple cases and gives a clean hook for character-class-diversity gates that RE2 cannot express directly (no lookaheads). Reusable for any future rule with the same shape.
- `go/internal/piiguard/builtin.go` line 34 — change the `bearer-token` `Pattern` to require `{20,}` on the captured token-value: `(?i)\bBearer\s+([A-Za-z0-9\-._~+/]{20,}=*)`. Set `Validator` to a function that returns false when the captured group consists entirely of lowercase ASCII letters (no digit, no uppercase, no punctuation). Real opaque values issued by spec-conformant servers are JWT-shaped or random base64 — both contain digits, uppercase, or punctuation. Keep the regex readable; the post-filter handles the corner case the regex cannot express.
- `go/internal/piiguard/scanner.go` — invoke `Rule.Validator` in `Scan()` immediately before appending to `Findings`. Single new line guarded by `if r.Validator != nil`. No behavior change for rules that don't set a Validator.
- `go/internal/piiguard/external.go` lines 23-39 — note that the three problem rules pass the `codingOnly` filter because the upstream `codingSecretKeywords` slice contains `"auth"`, `"token"`, and `"oauth"`, which match the rules' descriptions even though their regex bodies are keyword literals. We do NOT change the keyword filter; that risks dropping legitimate rules. Instead: add a package-level `disabledRuleIDs = map[string]bool{...}` containing the three slugified IDs from the editing convention, and skip them inside `loadExternalFromBytes` after the `codingOnly` filter, before `regexp.Compile`. Emit a stderr log `piiguard: skipping disabled rule %q` for visibility. Add a comment block above `disabledRuleIDs` explaining why each ID is off (literal-keyword matcher, not a credential pattern) so the next maintainer doesn't re-enable them as "looks useful."
- `.claude/rules/deleted-concepts.yaml` — add the three slugified IDs as deleted-concept entries with rationale, per `.claude/rules/migration-policy.md` Rule 1 (must ship in the same PR). The residue scanner will then catch any future PR that re-introduces them without justification.
- `go/cmd/harness/hook_pii.go` — introduce `piiguardWarnOnlyForEvent(event string) bool` returning true if `HARNESS_PIIGUARD_WARN_ONLY` is set OR the event-specific var (`HARNESS_PIIGUARD_PROMPT_WARN_ONLY`, `HARNESS_PIIGUARD_PRETOOL_WARN_ONLY`, `HARNESS_PIIGUARD_POSTTOOL_WARN_ONLY`) is set. Replace the existing `piiguardPromptWarnOnly()` callsite with `piiguardWarnOnlyForEvent("prompt")`. In `piiPreToolHandler`, on warn-only emit `permissionDecision: "allow"` with a `permissionDecisionReason` warning instead of `deny`, plus a stderr advisory and an `additionalContext` payload mirroring the post-tool warn shape (per advisor: PreToolUse warn means "model sees the warning post-hoc"). In `piiPostToolHandler`, on warn-only write only a stderr advisory and skip the `additionalContext` block.
- `go/internal/piiguard/testdata/negative/` — add four new fixture files: `bearer-prose.txt` (markdown describing the auth scheme + short header examples with 1-3 char captured values), `aws-terraform-doc.txt` (terraform-style snippet referencing AWS field names — both hyphenated and the literal underscore form, the latter validates 85.2 took effect), `oauth-doc-urls.txt` (the literal MS OAuth endpoint URLs as they appear in tutorials), `pwcrack-tool-prose.txt` (security writeup referencing the named password-cracking tool plus its known marker strings). The existing `TestCorpus_Negative` runs them automatically.
- `go/internal/piiguard/testdata/positive/` — add `bearer-real-token.txt` with a JWT-shaped Authorization header (3 dot-separated base64 segments, total ≥ 100 chars, contains digits and uppercase). Per advisor: required to prove task 85.1 didn't silently disable the `bearer-token` rule. The existing `TestCorpus_Positive` runs it automatically.
- `harness/CHANGELOG.md` — under `[Unreleased]`, add a Before/After entry per `.claude/rules/github-release.md` describing the user-visible noise reduction. Do NOT bump VERSION here — that happens at release time via `release-this`.

**Non-goals**:
- Adding a `.privacy-guard.json`-style per-rule user config file (deferred; warn-only env vars are the v1 escape hatch).
- Rewriting the external-catalog `codingOnly` filter — keyword-match heuristic is upstream-compatible behavior; only the three named rules are problem cases. If more weak rules surface, add IDs to `disabledRuleIDs`, don't redesign the filter.
- Touching the `openai-api-key` regex — already fixed in 74aebfa.
- Adding entropy-based heuristics. Per advisor: `{20,}` plus the all-lowercase-alphabetic post-filter is enough for v1; a Shannon-entropy gate is a separate, larger phase.
- Removing the three rule entries from the JSON catalog. We disable them via ID list, leaving the JSON intact so users with a different threat model can re-enable by clearing `disabledRuleIDs`.
- Modifying `harness/hooks/hooks.json`. Same hook wiring; only handler logic and rule loading change.
- Touching the `openai-api-key` positive fixture in `builtin_test.go`. Already realistic-length post-74aebfa.

**Risk notes**:
- **Post-filter false-negatives**: an attacker leaking an opaque token that is purely lowercase-alphabetic (rare but possible with custom hex-only or base32 tokens) would slip through. Mitigation: this is an acceptable trade-off; real opaque values issued by spec-conformant OAuth2 servers (RFC 6750) are JWT-shaped or random base64 — both contain digits or uppercase. The regex still catches everything ≥ 20 chars that has any digit/upper/punctuation. If a real token format ever surfaces that's purely lowercase, add a per-issuer rule.
- **Disabled-by-default removes a layer**: if the three context rules ever caught a real leak in some user's environment, disabling them weakens defense. Counter-evidence collected this session: the patterns are literal field names, URLs, and tool names — none represent secret values. They produced noise without finding secrets. Mitigation: keep the JSON entries in the catalog; users can re-enable by clearing `disabledRuleIDs`.
- **Warn-only env-var sprawl**: four env vars (global + three per-event) is a lot of surface. Mitigation: document the global `HARNESS_PIIGUARD_WARN_ONLY` first; present per-event vars as advanced tuning. Tests cover all four to catch precedence regressions.
- **Corpus-test self-fulfilling**: the negative fixtures assert zero findings under the FULL active catalog (builtin + external coding-only). Adding a new aggressive rule in the future will fail this test until the rule is tightened or the fixture is updated — that's the intended gate, not a bug. Document it in the first comment line of each fixture.
- **Fixture-write itself blocks**: writing `testdata/negative/bearer-prose.txt` via Edit/Write triggers the pre-tool-use PII Guard hook because the fixture content is, by design, the trigger strings. Mitigation: implementer sets `HARNESS_PIIGUARD_DISABLED=1` in their session before writing fixtures, OR writes them via a here-doc shell script that already runs in a hook-exempt context. Note this in the task DoD so the worker doesn't blame the hook.
- **Migration-policy compliance**: per `.claude/rules/migration-policy.md` Rule 1, deleting a concept (here: making rules inactive) must come with a `deleted-concepts.yaml` entry in the same PR. Task 85.2 covers both halves; the residue scanner check in task 85.6 verifies retroactively (Rule 4).
- **Editing-convention compliance**: any future edit to this Phase 85 block that expands the four rule names verbatim will fail the pre-tool-use PII Guard hook. The convention banner at the top of this phase exists to prevent that. After this phase ships and the rules are tightened, the banner can be relaxed in a follow-up edit.

## Phase 84: Port upstream v4.4.0 — sandbox network deny + AskUserQuestion normalizer

Created: 2026-04-28

**Goal**: Port three discrete changes from upstream `Chachamaru127/claude-code-harness@v4.3.3..v4.4.0` that are verified missing from the local fork (currently v4.12.1). (1) Block cloud-metadata service endpoints in `harness/settings.json` to prevent SSRF-style data exfiltration during sandboxed tool runs. (2) Add a Go-native AskUserQuestion hook handler that canonicalizes free-form workflow-mode answers (e.g. "single", "個人" → `solo`; "issue", "github-issue" → `team`; "explore" → `exploratory`) so downstream skills see one stable vocabulary. (3) Wire that handler into `harness/hooks/hooks.json` as a new PreToolUse matcher.

**Motivation**: The local fork is many releases ahead of upstream (v4.12.1 vs v4.4.0) but still missed three substantive items in the v4.4.0 cut: a security hardening (cloud metadata blocklist), a UX/normalization handler (AskUserQuestion canonicalizer), and the hook wiring that activates it. The metadata blocklist closes a known attack surface: any sandboxed shell that can reach 169.254.169.254 / metadata.google.internal / metadata.azure.com can read IAM credentials in seconds. The AskUserQuestion normalizer prevents downstream agents from receiving inconsistent mode strings — a real source of branch logic divergence between session-control, breezing, and team-composition skills.

**Source upstream**: Diff range https://github.com/Chachamaru127/claude-code-harness/compare/v4.3.3...v4.4.0. Specifically: (a) `.claude-plugin/settings.json` hunk adding `sandbox.network.deniedDomains`, (b) `go/internal/hookhandler/ask_user_question_normalizer.go` (326 lines) + its `_test.go` (209 lines), (c) `.claude-plugin/hooks.json` hunk adding the AskUserQuestion matcher. The upstream `valid_root()` resilience wrapper (which also appears in the same hooks.json diff) is intentionally **not** ported — the local plugin uses a different distribution layout (`harness/` subdir, no `.claude-plugin/plugin.json`) and our `${CLAUDE_PLUGIN_ROOT}` wiring already works reliably. Porting the wrapper as-is would either no-op (name check fails for "powerball-harness") or require extensive adaptation with no clear local benefit.

**Design**:
- `harness/settings.json` — append a single `network` block under the existing `sandbox` object; preserve `failIfUnavailable` and `filesystem` keys exactly as-is. Three deny entries, no allowlist.
- `go/internal/hookhandler/ask_user_question_normalizer.go` — port verbatim from upstream with two adaptations: (1) the canonical-value map keeps Japanese aliases since they are user input data, not code/comments; (2) any package-level identifiers must match local naming conventions in `go/internal/hookhandler/` (verify against neighboring `auto_cleanup_hook.go` and `elicitation_handler.go` for style).
- `go/internal/hookhandler/ask_user_question_normalizer_test.go` — port verbatim; the test suite exercises both English and Japanese alias paths plus pass-through for unknown values.
- `go/cmd/harness/hook.go` (or wherever the `hook` subcommand router lives) — register a new subcommand name (proposed: `ask-user-question-normalize`) that dispatches to the new handler. Match the existing dispatcher style (see how `pii-prompt`/`pii-pretool` are registered from Phase 83).
- `harness/hooks/hooks.json` — append a new PreToolUse entry with `matcher: "AskUserQuestion"` calling `"${CLAUDE_PLUGIN_ROOT}/bin/harness" hook ask-user-question-normalize` with `timeout: 3`. Do not modify existing hook entries (i.e. do NOT port the valid_root() wrapper; see Source upstream rationale).

**Non-goals**:
- Porting the `valid_root()` resilience wrapper to existing hook commands. Local plugin layout differs from upstream and the wrapper would need to be rewritten, not ported. If hook robustness becomes an actual problem locally, address it in a separate phase with a design that fits our layout.
- Porting upstream `sync.go` whitespace/alignment changes — cosmetic only, no functional impact.
- Porting upstream `setup-hook.sh`, `setup-existing-project.sh`, `templates/AGENTS.md.template` — local already has equivalent functionality (`hookhandler/setup_hook.go`, etc.).
- Renaming the canonical workflow-mode values (solo/team/exploratory) — those are a contract with downstream skills; this phase only normalizes inputs, never changes output vocabulary.
- Adding new aliases to `askQuestionCanonicalValues` beyond what upstream ships. Defer alias additions to a follow-up phase if needed.

## Phase 83: Native Go PII Guard (port claude-privacy-guard into the Go engine)

Created: 2026-04-28

**Goal**: Replace the Node.js scripts in [claude-privacy-guard](https://github.com/datumbrain/claude-privacy-guard/tree/main/scripts) with a native Go implementation under `go/internal/piiguard/`, exposed as new `bin/harness hook pii-*` subcommands. Wire it into `harness/hooks/hooks.json` so PII / secret detection runs on the user prompt (UserPromptSubmit), tool inputs (PreToolUse: Bash | Read | Write | Edit), and tool outputs (PostToolUse: Bash | Read). On a finding, block via Claude Code's hook decision API — no opt-in scanning, no in-place redaction. Timeout per hook: 2 seconds.

**Motivation**: claude-privacy-guard ships a Node.js UserPromptSubmit hook that blocks prompts containing 90+ secret/PII regex matches. It only covers user prompts, requires Node.js at runtime, and adds another binary to the dependency surface. Harness already runs a Go-native hook engine — folding this capability into `bin/harness` removes the runtime dependency, broadens coverage from "user prompts only" to "every tool I/O", and gives us one place to maintain the regex catalog. This closes the symmetric gap with the existing `go/internal/guardrail/` engine: guardrail covers tool *operations* (R01–R13: file paths, shell commands, tampering); piiguard covers tool *content* (secrets, PII).

**Source upstream**: Reference implementation at `github.com/datumbrain/claude-privacy-guard` MIT-licensed. Specifically: `scripts/prompt-guard.js` (decision flow), `src/scanner/engine.ts` (PrivacyScanner class), `src/scanner/detectors.ts` (BUILTIN_RULES + `loadExternalRulesFromJson`), `data/regex_list_1.json` (90-pattern external catalog). All to be re-implemented in Go, not vendored as JS.

**Design**:
- `go/internal/piiguard/` — new package: `Rule` struct (id, title, severity ∈ {critical,high,medium,low}, category ∈ {secret,pii}, pattern *regexp.Regexp), `Scanner` (slice of compiled Rules + `Scan(text) ScanResult`), `ScanResult` (Findings, RiskScore, Summary by category).
- `go/internal/piiguard/builtin.go` — port BUILTIN_RULES: email, JWT/Bearer, GitHub/HF tokens, AWS / OpenAI / Anthropic / Google / Groq / Perplexity / Stripe keys, SSH/RSA/EC private keys.
- `go/internal/piiguard/external.go` + `data/pii-regex.json` — port the 90-pattern external catalog; loaded with `codingOnly: true` filter (matches upstream default).
- `go/internal/piiguard/scanner.go` — `Scan(text)` returns findings, weighted risk score (critical=100, high=50, medium=20, low=10, capped at 100).
- New cmd handlers in `go/cmd/harness/hook_pii_*.go`: `hook pii-prompt` (UserPromptSubmit), `hook pii-pretool` (PreToolUse), `hook pii-posttool` (PostToolUse). Reuse one shared `runPIIGuard(input, blockMode)` helper; the three commands differ only in how they read the input and how they format the block decision.
- **Block protocol per event** (matches CC hook decision API):
  - UserPromptSubmit → `{"decision": "block", "reason": "🛡️ Privacy Guard blocked …"}` + exit 1 (mirrors claude-privacy-guard exactly)
  - PreToolUse → `{"hookSpecificOutput": {"hookEventName": "PreToolUse", "permissionDecision": "deny", "permissionDecisionReason": "…"}}` + exit 0 (CC v2.1.77+ deny semantics)
  - PostToolUse → `{"decision": "block", "reason": "…", "additionalContext": "<redacted output>"}` + exit 0 (CC PostToolUse cannot undo, but `decision:block` feeds reasoning back to Claude with a redacted view of the leaked data)
- **Hook config**: append three new entries to `harness/hooks/hooks.json` — UserPromptSubmit (matcher `*`), PreToolUse (matcher `Write|Edit|MultiEdit|Bash|Read`), PostToolUse (matcher `Bash|Read`). All `timeout: 2`.

**Non-goals**:
- Hooking PreToolUse `Task` / MCP tool calls (out of scope, separate phase if desired).
- Custom user allowlist UX — `.privacy-guard.json` config file is a follow-up; v1 ships with a hard-coded `disabledRules` env var (`HARNESS_PIIGUARD_DISABLED_RULES`).
- Vendoring or transpiling the upstream JS (Go re-implementation only).
- Replacing the existing 13-rule guardrail engine (piiguard runs alongside it).
- Removing the existing claude-privacy-guard plugin from a user's setup (docs will note the overlap; users can keep both, but Harness's piiguard is the recommended path inside this plugin).

## Phase 79: Explicit Failure Taxonomy (borrowed from NL-Agent-Harnesses paper)

Created: 2026-04-20

**Goal**: Consolidate scattered failure-handling logic into a single SSOT catalog, inspired by the "Failure Taxonomy" element (one of 8 NLAH elements) in *Natural-Language Agent Harnesses* ([arXiv:2603.25723v1](https://arxiv.org/abs/2603.25723)). Failure handling is currently spread across Go tampering patterns T01–T12 (`go/internal/hookhandler/post_tool.go`), Advisor error signatures (`harness/agents/advisor.md`), the Worker 3-retry-then-reticket loop (`harness-work` Phase 13), and CI fixer rules. Stable IDs (`FT-*`) give agents a shared vocabulary for detection, recovery, and escalation — and compose naturally with Phase 72 traces (trace events can cite taxonomy IDs for richer post-hoc audit).

**Paper prescription**: "Named failure modes drive recovery strategies." Turning tacit practice into an inspectable, referenceable artifact improves auditability and enables future ablation study (paper RQ2).

**Depends on**: Phase 72 (trace events in `.claude/state/traces/<task_id>.jsonl` are the richest emission point for `FT-*` IDs; 79.2 wires them in).

**Non-goals**: Replacing advisor history duplicate suppression, tampering detection logic, or CI fixer rules — only naming and cross-referencing them.
