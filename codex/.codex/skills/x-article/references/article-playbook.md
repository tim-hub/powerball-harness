# X記事プレイブック

このファイルは、X 記事を「読まれる」「最後まで読める」「共有されやすい」形にするための進め方です。

## 目次

1. 事前整理
2. タイトル設計
3. 記事構成
4. 本文ライティング
5. インプレッションを取りやすくする工夫
6. 出力フォーマット

## 1. 事前整理

記事を書く前に、次の 5 つを 1 行で埋める。

- **テーマ**: 何について書くか
- **読者**: 誰に向けて書くか
- **約束**: 読後に何を持ち帰れるか
- **根拠**: 何を証拠として示せるか
- **行動**: 読後に何をしてほしいか

例:

- テーマ: Claude Code の更新を Harness でどう増幅したか
- 読者: AI コーディングに関心がある開発者
- 約束: 単なる追従ではなく、実運用で何が強くなったかを理解できる
- 根拠: CHANGELOG、実装差分、テスト結果
- 行動: リポジトリを見に行く、感想を返す、次回更新を追う

## 2. タイトル設計

X 公式は「具体性」「好奇心」「価値の約束」を重視している。タイトル候補は最低 3 本出す。

### タイトルの型

1. **なぜ型**
   - 例: `Claude Code の更新をそのまま使うだけでは足りない理由`
2. **ビフォーアフター型**
   - 例: `Claude Code の 10 を、Harness で 15〜20 に引き上げた話`
3. **具体化型**
   - 例: `Claude Code 2.1.80〜2.1.84 を Harness に組み込んで変わった 5 つのこと`
4. **誤解修正型**
   - 例: `“追従しただけ” ではない。Harness で本当に効いた更新だけを拾った`
5. **読者便益型**
   - 例: `AI コーディングの更新を、実運用の強さに変える方法`

### タイトルの判定基準

- 何の記事かわかる
- 読む理由がある
- 抽象語だけで逃げていない
- 強すぎる煽りになっていない
- ヘッダー画像と並んだときに意味が重複しすぎない

## 3. 記事構成

基本構成は次を使う。

### 推奨構成

1. **冒頭フック**
   - 1〜3 段落
   - 読者の認識を止める
   - 何が変わったのかを端的に示す
2. **背景**
   - なぜこの話題が重要か
   - 今回の前提や観点
3. **本題セクション 3〜5 個**
   - 1 セクション 1 主張
   - 各セクションに具体例か比較を入れる
4. **総括**
   - 何が一番効いたか
   - 読者にとっての意味
5. **締め**
   - 一言でまとめる
   - 質問または行動の促しを置く

### セクションの書き方

- 小見出しは 3〜5 段落ごとに入れる
- 段落は 2〜4 行で切る
- 1 段落 1 アイデア
- 数字、固有名詞、比較をできるだけ入れる
- 箇条書きを適切に使う
- 大事な洞察は太字にする

## 4. 本文ライティング

### 書き出し

最初から説明しすぎない。まず「読む理由」を作る。

悪い例:
- `今回は Claude Code のアップデートについて説明します。`

良い例:
- `Claude Code が強くなっただけでは、現場の開発はそこまで変わりません。Harness に接続して初めて、更新が運用の強さになります。`

### 説明の仕方

- 主張をしたら、すぐに理由を置く
- 理由を置いたら、できれば実例を置く
- 専門用語は最初に短く意味を足す
- 長い列挙は表や箇条書きに逃がす

### 仕上げの編集

書き終えたら、最低でも次を行う。

- 冗長な接続詞を削る
- 同じ意味の繰り返しを削る
- 1 文が長すぎる箇所を分ける
- セクションの重複を削る
- 冒頭 5 行だけを読んで、続きを読みたくなるか確認する

## 5. インプレッションを取りやすくする工夫

ここは **Harness の提案** です。X 公式の原文ではなく、実務上の伸ばし方として扱う。

### 記事の中でやること

- 冒頭に **ズレ** か **意外性** を置く
- 具体的なバージョン、数字、固有名詞を入れる
- 途中に画像を入れて、スクロールを止める場所を作る
- 読者が引用しやすい一文を 1〜2 個入れる
- 締めで質問を置き、返信しやすくする

### 公開時にやること

- 予告ポストを出す
- 記事公開後 24〜72 時間はプロフィールに固定する
- 記事の抜粋を 3〜5 本の短いポストに分ける
- 初動の返信には早く返す

### 避けること

- タイトルだけ強くて本文が薄い
- 見出しが多すぎて要点が埋もれる
- 画像が本文と関係ない
- 一般論だけで終わる
- CTA がなく、読後に動けない

## 6. 出力フォーマット

### `out/social/<slug>/article.md`

最低限この順で保存する。

1. タイトル候補
2. 採用タイトル
3. リード
4. 本文
5. まとめ
6. 質問または CTA
7. 公開メモ

生成途中の package は、本文も画像も同じ `out/social/<slug>/` にまとめる。

`docs/social/` へ移すのは、公開用に正本化すると決めた後でよい。

### `out/social/<slug>/image-prompts.md`

最低限この順で保存する。

1. ヘッダー画像プロンプト
2. 差し込み画像プロンプト一覧
3. alt テキスト
4. 採用理由

### `out/social/<slug>/quality-report.md`

最低限この順で保存する。

1. 記事チェック結果
2. 画像チェック結果
3. 修正した点
4. 残る注意点
